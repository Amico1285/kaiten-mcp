MCP-сервер для **Kaiten** — 63 инструмента: карточки, комментарии, чеклисты, учёт времени, участники, блокеры, спринты, пользовательские свойства, внешние ссылки, файлы, история перемещений и глобальный тайм-шит.

Подключите Cursor, Claude Desktop, Claude Code или любой MCP-клиент к вашему Kaiten.

Запуск одной командой: `npx -y kaiten-mcp`.

Это форк [iamtemazhe/mcp-kaiten](https://github.com/iamtemazhe/mcp-kaiten) с большим объёмом доработок Wave 1–5: 22 новых инструмента, исправления схем, кросс-ресурсные preflight-проверки, упрощение ответов под LLM и улучшения надёжности. См. [WHYIEXIST.ru.md](WHYIEXIST.ru.md) и [CHANGELOG](../CHANGELOG.md).

## Быстрый старт

### 1. Получите API-токен

1. Откройте ваш Kaiten (например, `https://your-company.kaiten.ru`)
2. Профиль → API Key
3. Создайте токен и скопируйте его

### 2. Добавьте в ваш MCP-клиент

```json
{
  "mcpServers": {
    "kaiten": {
      "command": "npx",
      "args": ["-y", "kaiten-mcp"],
      "env": {
        "KAITEN_API_TOKEN": "your-api-token",
        "KAITEN_URL": "https://your-company.kaiten.ru"
      }
    }
  }
}
```

Готово. Сервер запускается автоматически при подключении клиента. `user_id` нигде указывать не нужно — текущий пользователь определяется по токену.

## Аутентификация

Kaiten использует API-токены. OAuth не поддерживается.

1. Профиль → API Key в вашем Kaiten
2. Создайте токен, скопируйте
3. Укажите `KAITEN_API_TOKEN` и `KAITEN_URL` в конфигурации MCP-клиента

Текущий пользователь определяется автоматически — `user_id` нигде указывать не нужно.

## Переменные окружения

**Обязательны только две переменные**: `KAITEN_API_TOKEN` и `KAITEN_URL`. Всё остальное опционально и существует для тюнинга производительности и безопасности.

| Переменная | Обязательна | По умолчанию | Описание |
|------------|-------------|--------------|----------|
| `KAITEN_API_TOKEN` | **да** | — | API-токен (Bearer) |
| `KAITEN_URL` | **да** | — | URL инстанса, например `https://your-company.kaiten.ru` |
| `KAITEN_DEFAULT_SPACE_ID` | нет | — | Пространство по умолчанию для `kaiten_search_cards`, если `spaceId` не задан. Без него LLM при первом поиске сам вызовет `kaiten_list_spaces` — лишний round-trip. Полезно если у вас много пространств и нужно зафиксировать default. |
| `KAITEN_REQUEST_TIMEOUT_MS` | нет | `10000` | HTTP-таймаут в миллисекундах |
| `KAITEN_CACHE_TTL_MS` | нет | `300000` | TTL для кэша справочных данных (пространства, доски, пользователи, роли) |
| `KAITEN_ALLOWED_SPACE_IDS` | нет | — | Белый список ID пространств через запятую (для multi-team safety) |
| `KAITEN_ALLOWED_BOARD_IDS` | нет | — | Белый список ID досок через запятую |

### Как узнать ID из URL браузера

Вам не нужны dev tools — каждый ID в Kaiten виден прямо в адресной строке.

- **Space ID** — кликните на пространство в левом меню, посмотрите URL: `https://your-company.kaiten.ru/space/762572` → space ID = `762572`.
- **Board ID** — кликните на доску, URL станет: `https://your-company.kaiten.ru/space/762572/boards/1727446` → board ID = `1727446`.
- **Card ID** — откройте карточку, URL: `https://your-company.kaiten.ru/space/762572/card/63258149` → card ID = `63258149` (также видно в шапке карточки как `#63258149`).

Эти значения подставляются в `KAITEN_DEFAULT_SPACE_ID`, `KAITEN_ALLOWED_SPACE_IDS`, `KAITEN_ALLOWED_BOARD_IDS`.

## Детализация ответов

Каждый инструмент принимает опциональный `verbosity` (по умолчанию `min`):

| Уровень | Описание |
|---------|----------|
| `min` | Компактный, экономит контекст LLM — только ключевые поля |
| `normal` | Основные поля — строгое надмножество `min` |
| `max` | Полная детализация — строгое надмножество `normal`, включает вложенных детей |
| `raw` | Сырой ответ API без преобразований |

Цепочка `min` → `normal` → `max` — это строгое надмножество (Wave 4 PR 4.6): всё, что видно в `min`, видно и в `normal`, и в `max`.

## Надёжность

- **Кросс-ресурсный preflight** (Wave 4 PR 4.7): мутирующие инструменты с парой parent/child ID (карточка + комментарий, карточка + чеклист и т. д.) проверяют принадлежность ребёнка родителю до выполнения мутации. Защита от silent cross-resource багов.
- **Author enrichment** (Wave 2): в ответах comment/timelog поле `author_name` подставляется client-side, даже если API вернул только `author_id`.
- **Контекстные подсказки в ошибках** (Wave 4 PR 4.5): к ошибкам API добавляется подсказка про связанный read-инструмент для восстановления (например, «404 на `/cards/{id}` → попробуй `kaiten_search_cards`»).
- **Автоматические повторы:** неудачные запросы (429, 5xx, сеть, таймауты) повторяются до 3 раз с экспоненциальной задержкой и джиттером. Заголовок `Retry-After` учитывается.
- **Идемпотентность:** write-запросы используют idempotency keys для защиты от дублирования при ретраях.
- **Кэш:** пространства, доски, пользователи, роли — в памяти с настраиваемым TTL. Устаревшие данные отдаются сразу, обновление идёт в фоне.
- **Защита от падений:** необработанные ошибки логируются без остановки сервера.
- **Усечение ответов:** очень большие ответы автоматически обрезаются.

## Особенности Kaiten API, о которых стоит знать

Реальные особенности Kaiten, обнаруженные в Wave 4–5. Каждая обработана в соответствующем инструменте, но LLM увидит их в описаниях.

- **`update_card.state` — read-only.** Поле `state` карточки вычисляется из `column.type` (1→queued, 2→in_progress, 3→done). Менять через `column_id`.
- **`update_card.size` не принимает число.** Используйте `sizeText: "5 SP"` (отправляется как `size_text`) или `estimate_workload` в секундах.
- **`owner_id` нельзя снять.** У карточки всегда есть owner — только переназначение.
- **Пустой `PATCH /cards/{id}`** возвращает 403 (странность API). Сервер ловит это до запроса с понятной ошибкой.
- **`DELETE` блокера — soft release.** Флаг `released:true` ставится, но строка остаётся в списке. Поэтому инструмент назван `kaiten_release_card_blocker`, а не `remove_*`.
- **`DELETE` внешней ссылки — настоящий hard-delete.** Асимметрия с блокерами — ссылка исчезает.
- **`location_history.id` — строка**, не число. Сохраняется как есть (парсинг в Number потерял бы точность для ID > 2^53).
- **Несуществующий sprint возвращает 403, не 404.** Оба статуса обрабатываются hint-хелпером.
- **Тайм-шит не принимает пустые array-фильтры.** `card_ids=` (пустое значение) даёт 400. Пустые массивы выпиливаются из query string.
- **Описания карточек и комментарии — markdown по умолчанию.** UI Kaiten рендерит description/comment как markdown. Если отправить чистый HTML без подсказки, угловые скобки останутся как есть и в UI будут видны буквально. Чтобы отправить HTML — передайте `textFormat: 'html'` в `kaiten_create_card` / `kaiten_update_card` / `kaiten_create_comment` / `kaiten_update_comment`. Тогда сервер распарсит и нормализует. Найдено live во время dogfooding-раунда 0.1.1 → 0.1.2 (cards используют документированный `text_format_type_id`; comments — недокументированный `type`, который мы проверили вживую).
- **Workspace-теги нельзя удалить через API.** Endpoint `/tags` поддерживает только `GET` и `POST` (проверено `docs/api/tags/`). `kaiten_remove_tag` снимает тег только с карточки, но не из workspace pool. Orphan-теги накапливаются и убираются только через админ-UI Kaiten.

## Решение проблем

- **Сервер не запускается:** проверьте, что `KAITEN_API_TOKEN` и `KAITEN_URL` заданы в `env` блоке конфигурации MCP.
- **Ошибки 401:** токен мог истечь или быть невалидным. Создайте новый в профиле Kaiten.
- **Большие ответы:** используйте фильтры (`boardId`, `spaceId`) или уменьшите `limit`.
- **403 на write-операцию:** проверьте, что карточка не в архиве и у токена есть права на запись в пространство.
