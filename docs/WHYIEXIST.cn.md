# 为什么选择 mcp-kaiten

## 问题

Kaiten 是常用的项目管理工具，但现有解决方案在 MCP 集成上各有取舍：

- 部分实现会根据 API 自动生成 246 个工具，严重挤占 AI 上下文
- 另一些侧重基础设施（日志、指标、诊断），但缺少工时、检查清单、标签和附件
- 还有一些对接 git 操作，却没有重试、超时和缓存
- 很少有替代服务器在工具数量可控的前提下覆盖完整的卡片生命周期

## mcp-kaiten 提供什么

### 1. 用 41 个工具覆盖完整卡片生命周期

围绕 Kaiten 卡片的完整工作流，同时避免压垮 AI 上下文：

**卡片** — 增删改查、15 种以上条件搜索、按空间或看板列出

**子任务** — 列出子卡片、挂载与解除子任务

**评论** — 列表、新建、更新、删除卡片评论

**工时** — 工时的完整增删改查，按用户或按卡片、任意日期范围查询

**空间与看板** — 浏览工作区结构：空间、看板、列、泳道、卡片类型

**标签** — 列出全部标签，为卡片添加或移除标签

**检查清单** — 创建、查看、删除检查清单；添加与更新清单项

**附件** — 列出、上传、删除卡片文件

**自定义字段** — 列出空间下可用的自定义属性

**用户** — 当前用户、用户列表、用户角色

### 2. 工具数量平衡

41 个工具足以应对真实场景，又不会淹没 AI 上下文。其他实现在域覆盖相近时可能暴露 246 个工具（例如由完整 API 自动生成）；基础设施向的替代服务器常见约 26 个工具，但缺少工时、检查清单、标签、附件和自定义字段。

### 3. 端到端的响应优化

整条请求—响应链路都为 AI 消费而优化：

- 只从 API 取必要数据，参考数据走缓存
- 四个详细程度级别，并对每种实体做类型化精简
- 自动截断超大响应，避免占满上下文
- 工具说明中包含工作流提示及相关工具的交叉引用
- 紧凑的输出格式节省 token

### 4. 可靠性

- 针对服务器错误、限流和网络故障的自动重试（指数退避 + 抖动）
- 防止在网络重试时出现重复提交
- 可配置请求超时
- 未处理错误下的崩溃防护
- 自动截断过大响应
- 可配置级别的结构化日志
- 可操作的错误信息，并提示下一步如何处理

### 5. 智能缓存

空间、看板、列、泳道、卡片类型、用户和标签在内存中按可配置的 TTL 缓存。可先立即返回略旧数据，同时在后台刷新 — 调用方不必等待缓存更新完成。

### 6. 访问限制

基于环境的限制，控制 AI 可访问哪些空间和看板 — 在多团队场景中，并非所有数据都应暴露给模型。

### 7. MCP 资源与提示

内置 MCP 资源可在不调用工具的情况下快速获取工作区结构（空间、看板）。提示模板引导常见流程：创建卡片、生成工时报表、查看看板。

### 8. 标准配置

环境变量在启动时做模式校验。无效配置会给出明确错误。支持任意基础 URL，适配私有化部署。

### 9. 体量小、依赖少

安装快、启动快、对外暴露面小。

### 10. 高级搜索

15 种以上筛选条件：标题、负责人、看板、列、标签、日期、卡片类型、完成状态、排除归档等。

## 优势

| 优势 | 说明 |
|---|---|
| 业务覆盖 | 工时、检查清单、附件、自定义字段、卡片类型、用户角色 |
| 工具数量 | 41 个工具，覆盖完整卡片生命周期 |
| 重试与防重复 | 应对服务器与网络问题，并避免重复生效的修改 |
| 访问限制 | 空间/看板白名单 |
| 智能缓存 | 即时响应 + 后台刷新 |
| 崩溃防护 | 未处理错误下的恢复能力 |
| MCP 资源与提示 | 内置工作区数据与工作流模板 |
| 依赖精简 | 2 个运行时依赖，原生 fetch |
| 私有化支持 | 任意 URL，不绑特定域名 |
| 扩展搜索 | 15 个以上筛选参数 |
| 结构化日志 | 可配置日志级别便于排查 |
| 可操作的错误 | 错误信息含下一步操作建议 |
